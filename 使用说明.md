# DiscordAITranslator 使用说明

## 1. 插件定位

`DiscordAITranslator` 是运行在 BetterDiscord 上的 AI 翻译插件，目标是：

- 让发送前翻译更顺手
- 让收到消息自动翻译更可控
- 让翻译保护规则尽量不破坏 Discord 原有消息结构
- 让译文插入、消息刷新和滚动恢复更稳定

## 2. 推荐配置顺序

1. 在“翻译服务”里选择主翻译服务商
2. 填入 API Key
3. 服务商需要时，再填写 Endpoint 和 Model
4. 在“语言设置”里配置发送 / 收到消息的输入与输出语言
5. 再去调整“显示与交互”
6. 最后按需设置“收到消息自动翻译”和“高级功能”

## 3. 发送消息翻译

### 发送前同语言跳过

`0.3.25` 起，发送消息前会先判断：

- 如果发送源语言和发送目标语言相同，直接发送原文
- 如果发送源语言固定为某语言，直接本地判定
- 如果发送源语言为“检测语言”，先本地检测再决定是否翻译

这样可以避免：

- 中文 -> AI -> 中文改写
- 英文 -> AI -> 英文改写

### 输入框总翻译开关

输入框右侧总翻译开关只控制“发送消息时是否自动翻译”，不再影响单条消息上的“翻译消息”按钮。

## 4. 收到消息自动翻译

### 同目标语言跳过

`0.3.26` 起，收到消息自动翻译前会先做本地预检测：

- 如果消息主体已经主要是目标语言，直接跳过
- 不调用 AI / 不调用翻译器
- 不应把同语言内容改写成另一句同语言内容

当前工作区还补充了一层缓存保护：

- 旧缓存里的同语言改写译文不会继续复用
- 消息重新渲染时，如果当前规则判定应跳过，会顺带清掉这类旧缓存

### 历史消息补翻

如果你开启已加载消息补翻：

- 插件会按当前已加载范围排队处理
- 会尽量复用合法缓存
- 会在消息刷新后恢复滚动位置

## 5. 手动翻译

### 当前消息正文优先

手动翻译会尽量只取当前消息正文：

- 不把引用预览正文混进去
- 不把引用里的剧透内容误当成本条消息正文

### 目标语言更严格

手动翻译模式下会更强调：

- 输出必须是设置的目标语言
- 不要根据原文里的双语内容或剧透内容误判目标语言

### 手动翻译滚动锁

点击旧消息“翻译消息”后：

- 插件会短时间锁定当前消息附近的滚动位置
- 新消息到来或译文插入后，尽量恢复到这条消息附近

## 6. 滚动与视角保持

### 自动翻译滚动锚点恢复

`0.3.26.scroll-anchor-manual-button-fix` 改成了“消息锚点恢复”：

- 刷新前记录当前正在看的 `messageId`
- 同时记录这条消息距离可视区域顶部的位置
- 刷新后重新找到同一条消息并恢复位置

这个逻辑主要用于减少：

- 自动翻译后跳到中间
- 新消息来了被拉到底部
- 历史消息补翻后视角错乱

## 7. 高级功能

### 短句不再默认跳过

`0.3.26` 移除了旧的短文本长度限制，像这些内容也应进入翻译流程：

- `嘻嘻`
- `哈哈`
- `嗯嗯`

### 剧透 `||` 处理

`||内容||` 不再被旧的自动保护包裹符规则直接挡掉。

这次修改只移除了 `||` 的普通包裹符保护逻辑，没有额外增加新的剧透总开关。

### 自动保护包裹符

适合成段保护内容，格式为：

```text
左包裹符|右包裹符
```

示例：

```text
`|`
【|】
「|」
"|"
```

### 保护词 / 保护短语

适合固定术语、产品名、团队名、专有名词，例如：

- `BUG team`
- `ChatGPT Plus`
- `Claude 3.7 Sonnet`
- `DeepSeek V3`

## 8. 推荐回归检查

1. 输入框总翻译开关关闭后，单条消息翻译按钮是否还能用
2. 收到中文消息且目标语言也是中文时，是否不再自动生成中文改写
3. 手动翻译外语消息时，是否稳定输出目标语言
4. `嘻嘻 / 哈哈 / 嗯嗯` 是否不再被漏掉
5. `||剧透内容||` 是否不再被普通包裹符规则拦住
6. 自动翻译后页面是否仍然保持在原来的观看位置

## 9. 修改后没有立刻生效

如果刚替换完插件文件或改完设置，但 Discord 没有立刻刷新，可以按：

```text
Ctrl+R
```

重新加载客户端。
